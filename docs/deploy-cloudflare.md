# Cloudflare deployment runbook

Step-by-step for taking Prisim R2 to a live Cloudflare Workers deployment
(via `@opennextjs/cloudflare`).

**Prerequisite**: get local working first — follow
[`local-dev.md`](./local-dev.md) end-to-end so you have:

- `pnpm` installed and `wrangler` already authenticated
- An R2 access token reserved for production (separate from your local one)
- A local `.dev.vars` that you've used to verify the full
  encrypt → decrypt → R2 probe loop

Everything below is _production-side_ configuration. Your local
`.dev.vars` is left untouched.

## 0. Generate production-only secrets

```bash
# Production AUTH_SECRET (Auth.js JWT signing)
openssl rand -base64 48

# Production ENCRYPTION_KEY (R2 credential AES-GCM master key)
openssl rand -base64 32
```

Stash both values in your password manager immediately. **Do NOT
reuse the values from your local `.dev.vars`** — sharing keys between
local and prod means a leak on either side compromises the other, and
local `.dev.vars` tends to slip into work journals or screenshots.

Treat both values as immutable. `ENCRYPTION_KEY` in particular is
essentially un-rotatable once in production — rotating means
re-encrypting every `connections` row. See "Rotating
`ENCRYPTION_KEY`" at the bottom of this doc.

## 1. Provision Cloudflare resources

The first deploy needs three things on the Cloudflare side: a Worker
slot (created automatically the first time `wrangler deploy` runs), a
D1 database, and the `name`/`database_id` written into `wrangler.toml`.

```bash
# Create the prod D1 database. Skip if it already exists on the account.
wrangler d1 create prisim-r2-db
```

Open `wrangler.toml` and confirm:

- `name = "prisim-r2"` matches the Worker you want to deploy. Change it
  if you want a different Worker name on Cloudflare.
- `[[d1_databases]]` `database_name` is `prisim-r2-db` and
  `database_id` is the UUID `wrangler d1 create` printed (or the
  existing one if you reused a database). Commit any change.

## 2. Set production secrets

Workers stores production env vars as Worker **secrets** (encrypted at
rest), set one-by-one via `wrangler secret put`. Run from the repo root
so wrangler reads `wrangler.toml`:

```bash
wrangler secret put AUTH_SECRET            # paste the 48-byte base64 from step 0
wrangler secret put AUTH_URL               # the Workers URL, e.g. https://prisim.example.com
wrangler secret put ENCRYPTION_KEY         # paste the 32-byte base64 from step 0
wrangler secret put NEXT_PUBLIC_APP_URL    # same value as AUTH_URL
```

Each command prompts for the value once. `wrangler secret list` lets
you confirm names are present without printing the values.

> `NEXT_PUBLIC_*` is read at build time, not runtime. After changing
> it you must redeploy (`pnpm deploy`) for the new value to take
> effect in the bundle.

> The D1 binding (`DB`) lives in `wrangler.toml` and is applied on
> every `wrangler deploy` — there's nothing to set in the Cloudflare
> dashboard for D1 itself.

## 3. Apply remote D1 migrations

```bash
pnpm db:migrate:prod
```

Applies every SQL file under `drizzle/migrations/` to the remote D1
database. **Run this only after the Worker secrets are set** — the
remote D1 state is independent of `.wrangler/state/`.

Re-run this command before every deploy whose diff includes a new
migration generated by `pnpm db:gen`.

## 4. Seed the production admin

```bash
ADMIN_EMAIL=you@example.com \
ADMIN_PASSWORD='<strong production password>' \
  pnpm tsx scripts/seed-admin.ts | tee /tmp/seed-prod.sql

wrangler d1 execute prisim-r2-db --remote --file=/tmp/seed-prod.sql
```

Note the `--remote` flag (not `--local`). The script requires
≥ 12 characters but doesn't gate on entropy — use a random string
from your password manager for production.

Delete `/tmp/seed-prod.sql` after applying. It contains a hashed
password, not plaintext, but there's no reason to keep it around.

## 5. Deploy

Once secrets, bindings, and remote migrations are in place, from your
local checkout:

```bash
pnpm deploy
```

This runs `opennextjs-cloudflare build` (compiling the Next.js app
into a Workers bundle under `.open-next/`) and then
`opennextjs-cloudflare deploy` (which wraps `wrangler deploy`). CI
deploys should run the same `pnpm deploy` — it's idempotent and safe
to invoke from GitHub Actions.

Sanity-check with a dry run before the first real push:

```bash
opennextjs-cloudflare build && pnpm exec wrangler deploy --dry-run
```

## 6. Apply per-bucket R2 configuration

For each production bucket the admin will browse/upload via Prisim:

1. **CORS** — add the Workers URL (e.g. `https://prisim.example.com`) to
   the bucket's `AllowedOrigins`. Full rules and the `wrangler` command
   are in [`r2-cors.md`](./r2-cors.md).
2. **Multipart lifecycle rule** — the `AbortIncompleteMultipartUpload`
   rule documented in [`multipart-cleanup.md`](./multipart-cleanup.md).
   Apply once per bucket that will receive uploads > 5 MB.

Without (1), uploads silently fail at the browser CORS preflight with
no useful error in the UI. Without (2), abandoned multipart parts
accumulate silently until the R2 bill makes it visible.

## Optional: run the E2E suite against a deployment

```bash
export E2E_BASE_URL=https://prisim.example.com
export E2E_ADMIN_EMAIL=you@example.com
export E2E_ADMIN_PASSWORD='<the production admin password>'
export E2E_R2_ACCOUNT_ID=<your 32-char R2 account id>
export E2E_R2_ACCESS_KEY=<R2 access key with read/write>
export E2E_R2_SECRET_KEY=<matching secret>
export E2E_R2_BUCKET=<a throwaway R2 bucket the spec can litter in>
export E2E_NO_WEBSERVER=1   # skip starting a local wrangler; we're hitting prod

pnpm test:e2e
```

Use a _separate_ admin user and bucket from your real workload — every
spec creates and deletes objects/connections/shares as part of running.

If you haven't installed Playwright's Chromium yet, run
`pnpm test:e2e:install` first.

## Rotating ENCRYPTION_KEY

Treat the key as immutable until you absolutely must rotate it. When
you do:

1. Generate a new key (`openssl rand -base64 32`).
2. Write a re-encryption script: for every `connections` row, decrypt
   with the old key and re-encrypt with the new one (the `id` ULID
   stays as AAD on both sides).
3. Apply the script while the production env still has the old key
   (a local utility connected to the remote D1, row-by-row UPDATE).
4. Swap the `ENCRYPTION_KEY` Worker secret on Cloudflare
   (`wrangler secret put ENCRYPTION_KEY`) to the new value.
5. Redeploy (`pnpm deploy`).

Skipping step 2 leaves every connection unreadable and silently breaks
every R2 call. There's no recovery path other than restoring the old
key — so keep a verified backup of the old key before starting step 2.
