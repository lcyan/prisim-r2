  统一错误格式

  所有错误返回：

  { "error": { "code": "string", "message": "human-readable", "requestId": "uuid" } }

  错误码分层

  | 范围 | 状态码 | 例子 |
  |---|---|
  | 客户端输入 | 400 | validation.invalid, confirm_token.mismatch |
  | 鉴权 | 401 / 403 | auth.required, csrf.missing |
  | 资源 | 404 | connection.not_found, object.not_found |
  | 限流 | 429 | rate_limit.exceeded |
  | 上游（R2） | 502 | r2.upstream_error（保留 R2 原始 code 作为子字段） |
  | 内部 | 500 | internal（永不暴露 stack/cred） |

  关键错误处理规则

  1. R2 凭证错误绝不回显原文。R2 返回 InvalidAccessKeyId / SignatureDoesNotMatch 时，统一映射为 connection.invalid_credentials，提示"请检查该
  connection 的凭证"，但不回显 access key id（即使被掩码）。
  2. 解密失败 = 数据完整性事件。如果 decryptCredential 抛错（tag 校验失败），写一条 audit_log.op='security.decrypt_failed'，返回
  connection.corupted，不允许重试。这通常意味着 ENCRYPTION_KEY 被换过或 DB 被篡改。
  3. 删除/危险操作的 confirm token。客户端先 POST /api/r2/delete/prepare 拿到一个 5 分钟有效的 confirmToken（HMAC(server-secret,
  user_id+bucket+keys_hash+exp)）。/api/r2/delete 校验 token 后才执行。这样防止 CSRF 之外，也防止前端 bug 误删。
  4. 限流错误带 Retry-After。429 响应必须含 Retry-After header（秒）。
  5. 每个错误都有 requestId。落审计日志和返回前端都用同一个 id，便于用户报障对账。

  每个 Route Handler 第一行：const input = Schema.parse(await req.json())。Zod schema 集中放在

  - 登录后随 session 一起下发 csrf token（存到 cookie + 通过 <meta> 暴露给 JS）
  - Same-Site=Lax 的 session cookie + 手动 token 双保险
